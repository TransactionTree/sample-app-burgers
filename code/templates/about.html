Getting Started

Getting started with NCR BSP APIs can be challenging when you first begin. This demo serves as an example as to how you can incorporate NCR BSP APIs in your application. This is a mock of a burger startup called NCR Burgers Unlimited. It is a Django/Bootstrap application. We have used Python to make the requests to NCR BSP API. All of the code snippets for this tutorial will be in Python. You can find relevant code snippets for other languages on the individual API pages.

Glossary

Application Programming Interface (API) - A piece of code is structured in such a way that allows other programs to use it without exposing the original code it was a part of. You can consider an API as a waiter. The API will bring you your order; you have no idea how the order was made.

Site - A site can be considered a physical location within the cloud. A site can be a restaurant, retail store, a warehouse, etc. This is the heart of the NCR BSP architecture. The site will be linked to many of the API calls that you make.

SiteId - A unique id generated by the siteAPI. This id is necessary for calls that depend on specific locations. Example: Retrieve the menu of XYZ location. XYZ location has a unique SiteId.

NEP-Correlation-ID - A unique debugging id (user-generated) to help map errors.

NEP-Organization - The top level organization that holds the information that is unique to a group of a sites. For example, The name of a group of sites. NCR-burgers might be the NEP-Organization, but there are three sites within the organization. Burgers Unlimited Southland, Burgers Unlimited Midtown, and Burgers Unlimited Highlands.

DevEx - Shorthand for the Developer Experience portal. It can be found here. It is the main resource for learning more about the various APIs available to you.

Payload - A command variable name for a formatted string containing relevant information. Payloads within BSP generally have a specific format that your strings must adhere to or your request will be denied

Sites

The site is the heart of the DevEx portal, the majority of the API calls within DevEx utilize a site. The first thing you should do is set up a site for your organization. You can have any number of sites, but you should at least create one. To create a site, you must utilize the Site API call. The documentation for the Site API can be found here.

Here is an example of creating a site using Python:

Now that we have our sites created let’s build on to that knowledge. We want to create a function that would allow us to display restaurants within a certain radius. For example, display all the restaurants within a certain radius of an input address. We can accomplish this easily using the find-nearby API call. Let's see an example of that: 

# Something worth noting is that this API call requires geocoordinates. You will need to determine how to transform an address into geocoordinates. There are several ways to do this.
 import requests

 def findResturantsInRange(coordinates, radius):
    # The radius in this function is in meters, so unit that user enters needs to be converted to from miles to meters. 1 mile = 1609 meters

    y = coordinates['y']
    x = coordinates ['x']
    circle = radius * meters

    url = 'https://gateway-staging.ncrcloud.com/site/sites/find-nearby/%s,%s' %(coordinates['y'],coordinates['x'])

    payload = {
        'numSites':10,
        'radius': circle,
    }

    r = requests.get(url, params=payload, auth=(HMACAuth()))

In this function, we are taking the input address coordinates and the input mile radius and searching for the sites that are within the radius. This will return a JSON object containing all of the restaurants in range. Your work is not done yet, because you will need to determine how to display this to the user. That will be determined by how you implemented your application.

However, this is not the only way to find restaurants based on distance. You could tweak this and instead of using find-nearby, you could use find-by-criteria. In which, the criteria would be the status = active. From there you could compile a list of restaurants and perform some calculations on how far each restaurant is from the given address. Your desired outcome will determine which APIs you will need to use.

Catalog

Now that we have given our restaurants a location, let’s talk about how to create a menu. To create a menu we will be utilizing the catalog API found here. The Catalog API views items and item components separately. The item is the base component. Once an item is created, you can mix and match the other components (details, custom-modifiers, groupings, prices, and link-groups) depending on your business need. Authentication

The APIs within BSP support Access Key (HMAC) and Access Token authentication methods. You should choose the authentication style that best fits your organization's needs and the intended users of your products. This sample app uses HMAC/Access Key authentication, which is discussed in further detail in the section below. For more information about how to use the other authentication styles, head over to the Authentication section in NCR API Platform Basics: Using Services.

HMAC

Access key authentication uses a hash-based message authentication code (HMAC) to uniquely sign a single HTTP request. This authentication method is intended for non-interactive users, e.g., technical users, involved in software-to-software or system-to-system authentication scenarios.

Access keys are comprised of two parts, a secret key and a shared key, and are managed by the 'security-access-keys' services in the 'security' application. Here is an example of access key authentication in a HTTP request: 

GET /provisioning/user-profiles HTTP/1.1
Accept: application/json
Accept-Encoding: gzip, deflate
Authorization: AccessKey e63ca6a9ca2e4db2bc13b741e7488437:Ysvt4LcqSnmIjvPbolVm2bS/zDXdqnYBtgtG+lWMlLI6uJp1MJiW34OVNtYrYA/B+6T/NDqhqFxbtlvuIFBliw==
Date: Wed, 26 Jun 2019 17:38:30 GMT
Host: gateway.ncrplatform.com
nep-application-key: 8a808f0d515f1f1001515f1fdc010002

The scheme used in the 'Authorization' header is 'AccessKey', and the credentials portion of the header consists of the following parts:

<shared-key>:<hmac>

Implementation

In each request, the credentials contain the user’s shared key and a HMAC generated from the user’s secret key. The general algorithm for generating the HMAC is as follows (see pseudocode here): 

1. Get the native DATE value from the request’s HTTP headers
2. Convert the native DATE to ISO-8601 string format
3. Generate a unique key from the following concatenated string: {secretKey + ':' + requestDate}
4. Generate a signature from the request data to use in HMAC calculation (see note below)
5. Generate HMAC using the SHA-512 algorithm, passing in the key and signature as inputs
6. Convert the outputted HMAC from above to Base64 Base64 encoding
7. Return the following concatenated string as the output: {sharedKey + ':' + encodedHMAC}

Note: A string is generated containing data from the request that will be used to generate a signature. When passed to the HMAC calculation, the string’s bytes should be encoded in UTF-8. The following data is added to the sign-able content (in required order, delimited by a '\n' character):

  • HTTP method from request (required)
  • Relative URI from HTTP request (path and query string) (required)
  • Content-Type header value (if specified in request)
  • Content-MD5 header value (if specified in request)
  • nep-application-key header value (if specified in request)
  • ep-correlation-id header value (if specified in request)
  • nep-organization header value (if specified in request)
  • nep-service-version header value (if specified in request)

Advantages

  • Allows maintenance of a single platform identity with multiple credentials expressed in the form of access keys (e.g., using one identity per application).
  • Individual access keys can be disabled or deleted without affecting other keys or the identity itself.
  • The secret key never expires, which simplifies interaction with the platform as compared to using temporary access tokens.
  • Guarantees authenticity of the full request instead of a specific part of the request (e.g., Authorization header).
  • Actual user secrets are never transmitted over the wire, as each request has a unique HMAC.
Here is the HMACAuth class and function used through the app:
(This is an example. This code snippet will not work without the correct environment and credentials.)

 import base64
 import hashlib
 import hmac
 from datetime import datetime
 from urllib.parse import urlparse
 import requests
 from _datetime import timezone
 from BurgersUnlimited.settings import *

 class HMACAuth(requests.auth.AuthBase):

    def __init__(self, enterprise_unit='0'):
        """
       Constructs all necessary attributes for the HMACAuth object.

       :param enterprise_unit: id of the nep enterprise unit the request applies to
       """
       self.enterprise_unit = enterprise_unit
       self.hmac_shared_key = HMAC_SHARED_KEY
       self.hmac_secret_key = HMAC_SECRET_KEY
       self.nep_application_key = NEP_APPLICATION_KEY

    def __call__(self, request):
       """
       Constructs HMAC to uniquely sign a HTTP authorization request.

       :param request: the HTTP request
       :return: the full credentials string for the HTTP Authorization header
       """
       now = datetime.now(tz=timezone.utc)
       now = datetime(now.year, now.month, now.day, hour=now.hour,
           minute=now.minute, second=now.second)

       isoDate = now.isoformat(timespec='milliseconds') + 'Z'
       utcDate = now.strftime('%a, %d %b %Y %H:%M:%S GMT')

       # Get the one-time key with the current date string
       key = self.customKey(isoDate)

       # Parse date string from header to a native representation
       parsedUrl = urlparse(request.url)

       # Get data from the request headers to sign in the HMAC string
       request.headers['Date'] = utcDate
       request.headers['Content-Type'] = 'application/json'

       if self.enterprise_unit != '0':
           request.headers['nep-enterprise-unit'] = self.enterprise_unit

       if 'site'  not in  parsedUrl.path:
           request.headers['nep-application-key'] = self.NEP_APPLICATION_KEY

       request.headers['nep-correlation-id'] = '2020-0708'
       request.headers['nep-organization'] = 'burgers-unlimited' 

       # Add the request data to the sign-able content
       self.addAuthorization(request, key)

       return request

    SECRET_KEY = HMAC_SECRET_KEY
    SHARED_KEY = HMAC_SHARED_KEY

    def customKey(self, date):
       """
       Generates a unique one-time key from the secret key and date string.

       :param date: the date string (ISO-8601 format)
       :return: a unique UTF-8 encoded key
       """
       key = self.SECRET_KEY + date
       return key.encode('utf-8')

    def addAuthorization(self, request, key):
       """
       Generates the HMAC signature for a HTTP authorization request.

       :param request: the HTTP request
       :param key: the access key string
       """
       parsedUrl = urlparse(request.url)
       pathAndQuery = parsedUrl.path

       if parsedUrl.query:
           pathAndQuery += '?' + parsedUrl.query

       # HTTP method and path/query are required parameters
       values = [request.method, pathAndQuery]

       # Add the HTTP header values to the sign-able content
       values.append(request.headers['Content-Type'])

       if 'site' not in pathAndQuery:
           values.append(request.headers['nep-application-key'])

       if 'nep-correlation-id' in request.headers:
           values.append(request.headers['nep-correlation-id'])

       values.append(request.headers['nep-organization'])

       separator = "\n"
       params = separator.join(values)

       # Convert the sign-able content to UTF-8 encoding
       encodedParams = params.encode('utf-8')

       # Calculate the HMAC using the SHA-512 algorithm
       hash = hmac.new(key, encodedParams, hashlib.sha512)
       digest = base64.b64encode(hash.digest()).decode('utf-8')

       # Concatenate the shared key and HMAC strings (UTF-8)
       accessKey = self.shared_key + ":" + digest

       # Add the signature to the HTTP authorization header
       request.headers['Authorization'] = "AccessKey " + accessKey
